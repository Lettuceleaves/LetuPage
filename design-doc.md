# LetuPage 设计文档

## 1. 项目定位

LetuPage 是一个面向 Agent 的前端框架，目标是在 Agent 运行环境、后端服务与前端页面之间建立一套稳定、可扩展、可复用的运行时体系。

第一阶段，项目聚焦于前端框架本身。框架不约束 Agent 或 CLI 侧的实现方式，而是提供标准 HTTP 接口，让前端页面把用户交互结果回传给后端，再由后端决定后续流程。

LetuPage 不只是一个组件库，而是一个包含以下内容的大框架：

- `main / session / page` 三层架构
- 会话与页面运行时
- 统一的页面生命周期与交互约定
- 封装好的样式体系与模块体系
- Agent 与页面通信模板
- 交互的打开、关闭、挂载、卸载等通用能力
- 图形资源库
- 组件资源库
- Skill 仓库接入约定

## 2. 核心目标

### 2.1 对 Agent 的目标

让 Agent 可以基于统一协议完成以下事情：

1. 创建或接入一个会话
2. 在会话中打开多个页面
3. 使用框架提供的页面基类和组件模块组织界面
4. 通过标准 HTTP 通道接收页面交互结果
5. 自主决定页面内容与交互逻辑
6. 安全地更新、关闭、替换页面

### 2.2 对框架开发者的目标

让开发者可以低成本扩展：

1. 新组件
2. 新页面基类
3. 新样式主题
4. 新资源包
5. 新 Skill 能力
6. 新运行时工具

### 2.3 对项目本身的要求

- 架构稳定，便于长期演进
- 协议清晰，便于后端与 Agent 对接
- 约定明确，避免页面行为失控
- 样式统一，但不束缚交互实现
- 扩展能力强，支持多类 Skill 与多种外部接入方式

## 3. 非目标

当前阶段不将以下内容作为第一优先级：

- 控制或规定 Agent 侧 CLI 的具体实现形态
- 对具体交互内容做强限制
- 用 JSON Schema 统一描述所有页面
- 将所有页面能力收敛为低代码配置
- 把图形资源、组件资源、Skill 资源混成同一套仓库结构

## 4. 总体设计原则

### 4.1 分层清晰

框架必须严格区分 `main`、`session`、`page` 三层职责，避免状态、视图、运行时逻辑混杂。

### 4.2 约束抽象，不约束创作

框架提供父类、接口、生命周期、通信规范和基础组件，但不限制 Agent 最终实现什么样的具体交互。

### 4.3 代码模块驱动

页面构建方式采用代码模块驱动，而不是 Schema 驱动。核心手段是：

- 用父类约束页面能力边界
- 用组件封装复用界面与交互单元
- 用模块注册机制组织可复用能力

### 4.4 多页面会话

一个 `session` 内允许同时存在多个 `page`。框架需要支持页面并存、切换、聚焦、更新和销毁。

### 4.5 Vue 3 优先

框架以 Vue 3 为基础技术栈，默认采用 Vue 3 生态能力构建运行时、页面基类和组件系统。

## 5. 架构总览

这里的“三层架构”指的是整个网页运行时的组织形式，不是仓库目录树，也不是资源仓库层级。

```text
网页应用
  -> main 层
    -> session 层
      -> page 层
```

对应关系如下：

- `main` 表示网页应用的全局入口层
- `session` 表示网页内部的会话运行时层
- `page` 表示网页中的页面实例层
- 三层架构描述的是网页如何组织和运行
- 图形资源库、组件资源库、Skill 仓库属于项目结构中的支撑模块，不属于这三层本身

## 6. 三层架构设计

## 6.1 Main 层

`main` 是框架入口层，负责全局级能力。

职责包括：

- 应用启动与初始化
- Vue 3 应用装配
- 全局路由或主容器管理
- 主题系统挂载
- 全局组件与模块注册
- 会话管理器初始化
- HTTP 交互接口注册
- 日志、调试、错误边界接入

不应承担的职责：

- 具体页面业务逻辑
- 具体页面交互实现
- 会话内部状态细节

## 6.2 Session 层

`session` 是任务级运行时边界，是页面交互结果与后端处理流程之间的主要承接层。

职责包括：

- `session` 创建、激活、销毁
- 多页面容器管理
- 页面注册与反注册
- 页面栈或页面集合维护
- 页面之间的通信调度
- 用户事件汇总与分发
- 会话级状态存储
- 会话级权限和能力注入
- 消息顺序控制与事件追踪

建议每个 `session` 至少维护以下对象：

- `sessionId`
- `taskId`
- `pages`
- `activePageId`
- `favoritePages`
- `eventBus`
- `sessionStore`
- `capabilities`
- `runtimeContext`

## 6.3 Page 层

`page` 是用户可见的最小页面单元。

职责包括：

- 页面实例创建
- 页面生命周期执行
- 页面局部状态维护
- 页面事件上报
- 页面 UI 结构组织
- 页面打开、关闭、隐藏、聚焦
- 页面与组件、Skill、资源的组合

一个 `page` 不直接承担会话调度职责，而是通过会话上下文与外部通信。

## 7. 页面模型设计

### 7.1 页面实例定义

建议每个页面实例具备以下基础属性：

- `pageId`
- `pageKey`
- `sessionId`
- `type`
- `title`
- `status`
- `visible`
- `closable`
- `favorited`
- `payload`
- `meta`
- `createdAt`
- `updatedAt`

### 7.2 页面基类

页面采用父类约束。页面基类负责定义统一的能力边界和默认行为。

建议页面基类至少提供：

- 初始化方法
- 挂载方法
- 更新方法
- 销毁方法
- 打开方法
- 关闭方法
- 显示/隐藏方法
- 聚焦方法
- 事件派发方法
- 接收外部消息方法
- 错误兜底方法

示意：

```ts
abstract class BasePage {
  abstract pageType: string

  onInit(): void {}
  onMount(): void {}
  onOpen(): void {}
  onClose(): void {}
  onUpdate(payload: unknown): void {}
  onDestroy(): void {}

  emit(event: string, payload?: unknown): void {}
  receive(message: unknown): void {}
}
```

这里的重点不是强制所有页面长得一样，而是统一页面运行时行为。

### 7.3 页面组织方式

页面采用代码模块驱动，主要包括两类元素：

- 页面类或页面实例控制器
- Vue 组件树

推荐模式：

1. 用页面基类定义运行时行为
2. 用页面级控制器连接会话和视图
3. 用 Vue 组件承载具体 UI
4. 用 Skill 承载复杂交互逻辑

## 8. 组件体系设计

组件资源库与图形资源库分离管理。

### 8.1 组件资源库定位

组件资源库用于提供可复用的 UI 单元和页面模块。

建议分层：

- 基础组件
- 页面结构组件
- 业务无关交互组件
- 结果展示组件
- 容器型组件

建议首批组件范围：

- 页面外壳
- 标题栏
- 收藏栏
- 页面导航栏
- 页面列表弹窗
- 面板
- 卡片
- 表单容器
- 输入包装器
- 按钮组
- 状态条
- 日志流
- 结果展示块
- 预览容器
- 工具栏

### 8.2 组件设计原则

- 尽量无业务含义
- 尽量组合优先
- 风格统一
- 状态清晰
- 支持插槽和扩展
- 不把复杂交互硬编码进框架核心

## 9. 交互能力设计

这是本项目最关键的边界之一。

LetuPage 框架只提供交互抽象与约定，不限制具体交互实现。也就是说：

- 框架可以提供交互父类
- 框架可以提供交互接口
- 框架可以提供通信规范
- 框架可以提供交互实例的打开、关闭、挂载、卸载约定
- 框架可以提供常见基础控件
- 具体交互如何实现，由 Agent 或交互 Skill 决定

### 9.1 HTTP 交互回传模型

当前阶段采用最简单的 HTTP 模式完成页面与后端的交互闭环。

基本流程如下：

1. 前端页面展示由后端或 Agent 产出的内容
2. 用户在页面中进行输入、选择、确认、取消等操作
3. 前端将交互结果通过 HTTP `POST` 提交给后端
4. 后端通过响应码和响应体告知前端下一步动作
5. 前端根据响应结果决定保持当前交互、关闭交互弹窗，或进入错误状态

这个模型的重点是：

- 页面负责采集用户交互
- 前端负责把交互结果标准化并发送给后端
- 后端负责接收结果并决定流程是否继续
- Agent 或 CLI 侧如何消费后端结果，不由前端框架约束

### 9.2 交互父类建议

建议定义一个统一的交互基类，用于约束交互模块的生命周期和通信方法。

例如：

```ts
abstract class BaseInteraction {
  abstract interactionType: string

  onInit(): void {}
  onAttach(): void {}
  onDetach(): void {}
  onSubmit(payload: unknown): void {}
  onCancel(): void {}

  emit(event: string, payload?: unknown): void {}
}
```

### 9.3 交互提交与响应约定

建议每个交互实例都具备统一的提交接口，至少应包含以下字段：

- `sessionId`
- `pageId`
- `interactionId`
- `action`
- `payload`
- `timestamp`

示意：

```ts
interface InteractionSubmitPayload<T = unknown> {
  sessionId: string
  pageId: string
  interactionId: string
  action: string
  payload?: T
  timestamp: number
}
```

建议后端响应至少包含两层语义：

1. HTTP 响应码
2. 响应体中的前端动作指令

建议前端默认按以下规则处理：

- `2xx`：表示后端成功接收并接受本次交互
- `4xx`：表示当前用户输入或交互状态存在问题，前端应保持交互打开并提示错误
- `5xx`：表示服务异常，前端应保持交互打开并进入失败提示状态

为了避免只靠响应码承载全部语义，建议响应体增加显式动作字段：

```ts
interface InteractionResponse {
  code: number
  action: "keep" | "close" | "refresh" | "redirect"
  message?: string
  payload?: unknown
}
```

推荐处理规则：

- `200` + `action=close`：关闭当前交互弹窗
- `200` + `action=keep`：保持当前交互打开
- `200` + `action=refresh`：更新页面或交互内容后保持打开
- `4xx` + `message`：显示校验错误并保持打开
- `5xx` + `message`：显示服务异常并保持打开

这样前端可以利用响应码区分成功与失败，再通过 `action` 字段决定具体 UI 行为。

### 9.4 不做的事情

框架不应：

- 强制所有交互都映射成固定 Schema
- 把所有交互限制成预定义控件集合
- 限制 Agent 必须按照单一表单范式组织页面

### 9.5 Skill 的角色

复杂交互能力由 Skill 承担，例如：

- 复杂表单 Skill
- 图片选择 Skill
- 多步骤流程 Skill
- 表格编辑 Skill
- 富文本或结构化输入 Skill

框架负责让 Skill 有稳定接入点，而不是替 Skill 实现全部业务。

## 10. 会话与多页面机制

一个 `session` 允许多个 `page` 同时存在，因此框架必须支持以下能力：

- 多页面注册
- 当前活动页切换
- 页面并存显示
- 页面隐藏与恢复
- 页面分组
- 页面间事件转发
- 页面批量关闭

建议会话层至少支持以下操作：

- `createPage`
- `mountPage`
- `focusPage`
- `hidePage`
- `showPage`
- `updatePage`
- `closePage`
- `destroyPage`
- `broadcast`

### 10.1 多页面展示策略

虽然允许多个页面并存，但建议在第一版中保留统一容器策略，例如：

- 主工作区页面
- 辅助侧栏页面
- 浮层页面

这样可以避免页面自由堆叠导致管理失控。

### 10.2 页面收藏机制

框架需要提供页面收藏能力，用于把常用页面固定在收藏栏中。

规则如下：

- 页面可以被加入收藏栏
- 收藏后的页面应固定显示在收藏栏
- 收藏信息必须持久化到 `localStorage`
- 收藏栏与普通页面导航栏分离
- 收藏持久化内容至少应包含 `pageKey`、`title`、`type`、必要的恢复元信息

设计约束：

- `pageId` 是会话内实例 ID，不适合直接作为跨刷新持久化主键
- 收藏持久化应基于稳定标识，例如 `pageKey`
- 如果某类页面支持恢复，则应提供可序列化的恢复信息
- 如果某类页面不支持恢复，则至少允许保留收藏入口描述

建议增加以下会话级或应用级能力：

- `favoritePage`
- `unfavoritePage`
- `loadFavorites`
- `saveFavorites`

### 10.3 页面导航排序规则

除收藏栏外，普通页面导航采用“最近修改时间优先”规则。

规则如下：

- 全部页面默认按照 `updatedAt` 倒序排列
- 最近被修改或被选中的页面，应自动更新 `updatedAt`
- 更新后的页面应移动到导航栏最前面
- 页面关闭后应从普通导航列表移除

这里的“修改”包括但不限于：

- 页面内容更新
- 页面被重新激活
- 用户从导航中再次选中页面

### 10.4 导航栏容量与溢出规则

当页面较多时，主导航栏不应无限展开。

建议规则如下：

- 主导航栏默认最多显示 10 个页面入口
- 这 10 个入口按 `updatedAt` 倒序显示
- 超出 10 个的页面不直接占用主导航栏宽度
- 主导航栏末尾提供一个“展开”按钮
- 点击“展开”按钮后，显示一个可上下滚动的小窗口
- 小窗口中列出当前会话全部页面
- 用户可以在小窗口中直接选择任意页面

交互要求如下：

- 在小窗口中选中某个页面后，该页面应立即成为当前活动页
- 该页面的 `updatedAt` 应自动刷新
- 刷新后该页面应进入主导航栏最前面
- 如果原本在主导航栏中的最后一项被挤出，应自动进入展开列表

### 10.5 导航结构建议

基于以上规则，页面导航建议拆成两部分：

1. 收藏栏
2. 最近页面导航栏

推荐布局：

- 收藏栏用于固定收藏页面
- 最近页面导航栏用于展示最近活跃的 10 个页面
- 展开弹窗用于展示完整页面列表

这样可以同时满足“固定入口”和“高频最近访问”的双重需求。

## 11. 通信协议设计

第一阶段需要先定义前端内部协议与外部 HTTP 接口草案，使页面交互结果可以稳定回传到后端。

### 11.1 协议目标

- 统一页面交互结果的提交方式
- 统一会话与页面相关的交互事件结构
- 为后端、Agent、CLI 等外部处理方提供稳定边界

### 11.2 消息结构建议

```ts
interface RuntimeMessage<T = unknown> {
  version: string
  type: string
  sessionId: string
  pageId?: string
  requestId?: string
  timestamp: number
  payload?: T
  meta?: Record<string, unknown>
}
```

### 11.3 HTTP 接口建议

建议至少提供一条统一交互上报接口：

```text
POST /api/session/:sessionId/page/:pageId/interaction/:interactionId
```

请求体用于承载交互结果，响应体用于告知前端后续行为。

如果后续需要扩展，也可以增加：

- `POST /api/session/:sessionId/page/:pageId/event`
- `POST /api/session/:sessionId/page/:pageId/favorite`
- `POST /api/session/:sessionId/page/:pageId/unfavorite`

### 11.4 建议消息类型

- `session.create`
- `session.destroy`
- `session.activate`
- `page.open`
- `page.update`
- `page.close`
- `page.focus`
- `page.hide`
- `page.show`
- `ui.event`
- `ui.submit`
- `ui.cancel`
- `skill.invoke`
- `skill.result`
- `runtime.error`

### 11.5 协议原则

- 统一 JSON 序列化格式
- 带版本号
- 带会话 ID
- 带页面 ID
- 带时间戳
- 保留请求关联字段
- 前端与后端的交互默认使用 HTTP `POST`
- 前端根据响应码和响应体动作字段决定后续 UI 行为

## 12. 样式系统设计

样式系统是框架的一部分，但不与图形资源库混为一体。

### 12.1 样式系统职责

- 设计令牌
- 颜色规范
- 排版规范
- 间距规范
- 边框与圆角规范
- 阴影规范
- 动效规范
- 页面布局规范

### 12.2 样式系统目标

- 保证整体视觉统一
- 降低页面开发成本
- 让不同 Agent 输出页面时仍保持项目感
- 支持后续多主题演进

### 12.3 样式边界

样式系统负责“怎么统一”，不负责“用什么图片”。图片、插画、背景素材等属于图形资源库。

## 13. 资源体系设计

项目资源需要拆分为三类，而不是混为一类。它们属于项目结构的一部分，通常位于 `src` 下的对应目录中，用来支撑三层架构运行。

## 13.1 图形资源库

图形资源库只负责视觉素材。

一个素材一个文件夹，建议结构如下：

```text
graphic-assets/
  clean-tech/
    README.md
    preview.png
  soft-panel/
    README.md
    preview.png
```

其中 `README.md` 建议包含：

- 素材名称
- 风格描述
- 关键词
- 适用场景
- 推荐搭配颜色
- 使用限制
- 推荐页面类型

图形资源库的职责：

- 为页面提供背景、插图、装饰图、氛围图
- 帮助 Agent 美化页面
- 通过标签和描述辅助检索

图形资源库不负责：

- 页面结构定义
- 组件逻辑
- Skill 能力实现

## 13.2 组件资源库

组件资源库存放 Vue 组件、页面模块和组合式 UI 能力。

建议结构如下：

```text
component-assets/
  base/
  layout/
  feedback/
  result/
  container/
```

组件资源库负责：

- 组件复用
- 样式一致性
- 页面拼装效率

## 13.3 Skill 仓库

Skill 仓库用于存放 Agent 可调用的扩展能力约定。

Skill 可覆盖：

- 页面生成辅助
- 交互增强
- 表单能力
- 内容排版
- 数据渲染
- 资源检索
- 特殊页面工作流

Skill 仓库与框架的关系：

- 框架提供接入点
- Skill 提供具体能力
- Skill 不应破坏框架生命周期和协议

## 14. 模块与扩展机制

为了让框架长期可维护，建议设计统一的注册机制。

### 14.1 可注册对象

- 页面类型
- 页面基类实现
- 组件
- 容器
- Skill
- 主题
- 工具
- 资源索引器

### 14.2 注册目标

- 明确依赖关系
- 避免隐式注入
- 便于调试
- 便于按需加载

### 14.3 扩展原则

- 扩展通过注册完成
- 扩展不得直接绕过会话层操作页面
- 扩展必须遵守统一协议

## 15. 建议项目结构

这一节描述的是 `src` 内的项目目录结构，不是网页的三层运行时架构。

以下是一个更符合当前设计的初版目录建议：

```text
src/
  main/
    app.ts
    bootstrap.ts
    registry/
  session/
    manager/
    runtime/
    store/
    protocol/
  page/
    base/
    controllers/
    registry/
    views/
  interaction/
    base/
    adapters/
  components/
    base/
    layout/
    feedback/
    result/
  styles/
    tokens/
    themes/
    motion/
  skills/
    registry/
    adapters/
  assets/
    graphics/
    icons/
```

其中：

- `main / session / page` 对应网页三层架构在代码中的落点
- `interaction / components / styles / skills / assets` 是支撑三层架构的项目模块
- 图形资源库、组件资源库、Skill 仓库都应先体现在 `src` 结构中

如果后续要把资源仓库外置，也可以拆成独立仓库再通过索引方式接入。

## 16. MVP 范围建议

第一阶段建议只做以下内容：

1. Vue 3 项目骨架
2. `main / session / page` 三层基础实现
3. 页面基类与交互基类
4. 多页面会话容器
5. 基础通信协议定义
6. 面向后端的 HTTP 交互提交接口
7. 页面收藏栏与 `localStorage` 持久化
8. 最近修改页面排序与导航溢出机制
9. 组件注册机制
10. Skill 注册机制
11. 样式令牌与基础主题
12. 图形资源库目录规范
13. 组件资源库目录规范

第一阶段暂不强行实现：

- 丰富业务 Skill
- 复杂页面编排器
- 可视化搭建器

## 17. 后续演进方向

第二阶段可以逐步补充：

- 更丰富的 HTTP 接入与事件回传适配器
- 页面持久化与恢复
- Skill 热插拔
- 资源检索索引
- 页面调试面板
- 协议录制与回放
- 多会话管理界面

## 18. 仍需继续讨论的问题

当前核心方向已经确定，但下面这些点后续还需要继续细化：

1. `session` 内多个页面的默认布局策略到底采用标签页、分栏、浮层，还是混合容器。
2. 页面基类与 Vue 组件之间的绑定方式采用继承包装、控制器注入，还是组合式 API 适配层。
3. Skill 的装载方式采用本地注册、动态加载，还是独立包机制。
4. 图形资源库和组件资源库是否放在同仓库下管理，还是拆分成独立仓库。
5. HTTP 交互接口的返回动作字段是否需要统一成固定枚举，还是允许业务扩展。
6. 页面交互上报是统一单接口，还是按事件类型拆分多个接口。

## 19. 结论

LetuPage 的第一阶段应当明确定位为一个基于 Vue 3 的 Agent 前端框架。

它的核心不是“把页面做死”，而是建立一套稳定的大框架：

- 用 `main / session / page` 管理运行时
- 用父类约束页面和交互能力
- 用组件封装界面
- 用 Skill 承载复杂交互
- 用 HTTP 回传用户交互结果，并通过响应码和响应动作控制前端状态
- 用图形资源库、组件资源库、Skill 仓库支撑扩展

在这个前提下，Agent 可以拥有充分的页面生成自由，但系统整体仍然保持可维护、可扩展、可演进。
