# Git 使用规范

## 1. 目标

本规范用于约束 LetuPage 项目的 Git 提交时机、提交粒度和 commit message 格式，确保提交历史清晰、可追踪、可回滚。

## 2. 提交时机

只有在以下条件同时满足时才应提交：

- 当前改动已经形成一个完整的逻辑单元
- 代码、文档、样式或配置已经处于自洽状态
- 相关验证已经执行，或能够明确说明未执行原因
- 没有未说明的临时代码、调试输出、占位实现或意外残留

对于多步骤任务，应按“稳定里程碑”提交，不应按编辑次数提交。

## 3. 不应提交的情形

以下情况禁止直接提交：

- 仍处于探索或试错阶段
- 改动尚未收敛
- 一个提交混入多个无关主题
- 验证失败且未说明原因
- 工作区中包含临时文件、生成噪音、密钥或敏感信息
- 存在未理解的他人改动

## 4. 提交粒度

提交必须遵循“单一目的”原则：

- 功能实现与重构分开提交，除非两者无法拆开理解
- 代码修改与文档修改分开提交；如果文档是该改动不可分割的一部分，可以同一提交
- 纯格式化与行为变化分开提交
- 大规模重命名与逻辑修改分开提交

如果一个提交无法用一句话准确描述，通常说明粒度过大。

## 5. 提交前检查

提交前至少执行以下检查：

- 查看 `git status --short`
- 检查待提交文件是否都属于当前任务
- 检查 `git diff --cached` 或等价摘要
- 确认是否混入无关文件
- 确认验证结果或记录未验证原因

如果工作区中有用户自己的改动或其他任务的改动，只暂存当前任务相关文件，不要擅自整理其他改动。

## 6. Commit Message 格式

统一格式如下：

```text
<type>(<scope>): <summary>
```

约束如下：

- `type` 使用小写英文
- `scope` 使用小写英文或 kebab-case
- `summary` 使用简短中文，聚焦结果，不写过程，不加句号

推荐 `type`：

- `feat`：新增功能
- `fix`：修复缺陷
- `refactor`：重构但不改变外部行为
- `docs`：文档变更
- `style`：纯样式或格式调整
- `test`：测试补充或调整
- `chore`：杂项维护、脚本、配置调整
- `build`：构建流程或依赖管理调整
- `ci`：持续集成配置修改
- `revert`：回滚已有提交

推荐 `scope`：

- `main`
- `session`
- `page`
- `interaction`
- `components`
- `styles`
- `assets`
- `skills`
- `docs`
- `git`

示例：

```text
feat(session): 增加多页面导航排序规则
fix(page): 修正页面收藏状态持久化
refactor(interaction): 抽离 HTTP 提交基类
docs(git): 补充提交时机与 message 规范
```

## 7. 执行规则

- 只暂存当前任务相关文件
- 用户未明确要求时，不主动 `git push`
- 用户未明确要求时，不执行 `git commit --amend`
- 用户未明确要求时，不改写历史，不 `rebase`，不 `squash`，不 `force push`

## 8. 项目特别约束

- 设计文档修改优先单独提交，不要和框架代码实现混成一个提交，除非文档与代码必须同步出现
- `main / session / page` 跨层改动按功能链路拆分，不按文件夹机械拆分
- 页面导航、收藏、交互协议等独立规则修改优先单独提交
- Skill 文件改动优先与对应规范文档一同提交，避免本地 skill 与仓库文档不一致

## 9. 提交后汇报

完成提交后，应至少记录：

- 提交 hash
- commit message
- 本次提交核心内容
- 已执行验证或未执行验证原因
